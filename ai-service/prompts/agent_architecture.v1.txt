## Review Instructions

### 1. Score FOUR categories independently (0–100 each)
- **correctness** — does the code do what it appears intended to do? Edge cases handled? Error paths sound? Off-by-ones, null/None handling, type coercion, boundary conditions.
- **readability** — naming, function length, indentation, comment quality, code organization, intent-revealing structure.
- **design** — separation of concerns, abstraction levels, coupling, layering, idiomatic patterns, testability, change-amplification risk.
- **taskFit** — does the code ACTUALLY IMPLEMENT what the task brief (Project Description above) asked for? Read the project description carefully — it carries the task title, description, acceptance criteria, and deliverables. Then judge whether the submitted code addresses those requirements specifically (not just whether it's good code in general).

### Task-fit grading rules (READ CAREFULLY)
- **High taskFit (80–100):** the submission clearly implements the task's main deliverable; the acceptance criteria are visibly addressed by named files / functions / endpoints in the code.
- **Medium taskFit (50–79):** the submission addresses part of the task or has the right shape but is missing one or more acceptance criteria.
- **Low taskFit (20–49):** the submission relates to the task domain but doesn't actually solve the stated problem (e.g., wrong feature scope, off-topic prototype, scaffolding only).
- **Very low taskFit (0–19):** the submission is unrelated to the task (e.g., a chat app submitted for a binary-search task, or copy-pasted boilerplate).
- **STRICT RULE:** even if the code is clean and well-architected (high correctness/readability/design), if it does not implement the task brief you MUST set taskFit ≤ 30 and prominently call this out in the FIRST weakness AND the executive summary's opening sentence. Do NOT give a high overall by being polite — the learner needs to know they delivered something different than what was requested.

### 2. Identify 3-6 architectural findings
For each, give exact file:line, the problematic snippet, an educational explanation, and a corrected example. These map to detailed issues in the merged response.

### 3. Identify 3-5 strengths and 3-5 weaknesses
Strengths/weaknesses are concise, learner-facing summary statements. The detailed versions go in `strengthsDetailed` / `weaknessesDetailed`.

### 4. Provide 2-4 prioritized recommendations
Specific, actionable next steps with priority and category.

### 5. Curate 1-3 learning resources per major weakness
Use reputable sources: official docs (MDN, Python docs, Microsoft docs), OWASP, freeCodeCamp, Real Python, Baeldung, official framework docs. No blog spam, no broken links.

### 6. Write executive summary (3-4 paragraphs) and brief summary (2-3 sentences)
Executive summary: overall assessment + notable strengths + critical attention areas + prioritized action items.
Summary: a quick-reference one-paragraph version.
