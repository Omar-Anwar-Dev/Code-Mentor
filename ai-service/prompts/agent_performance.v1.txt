## Review Instructions
1. Examine every file for performance-impacting patterns: nested loops with shared scaling, N+1 database queries, blocking I/O in async/await contexts, repeated computation that could be memoized, large allocations inside hot loops, inefficient data-structure choices (list-as-set lookups, dict construction in tight loops), unnecessary serialization round-trips, missing pagination/streaming, regex re-compilation, string-concatenation in loops, eager evaluation where lazy would do.
2. For each genuine finding, give the EXACT file path AND line number, the problematic snippet, the complexity / latency / allocation characterization (concrete — "this is O(n²) where n = number of users; at 10k users this becomes ~100M operations"), and a complete corrected example.
3. Score `performanceScore` 0–100 using this rubric:
   - 90–100: no realistic bottlenecks at expected scale; data structures and algorithms are appropriate; async correctly handled.
   - 70–89: 1–2 medium findings (e.g., minor extra allocations, suboptimal data structure for non-hot path).
   - 50–69: 1 high-impact finding (e.g., N+1 query, O(n²) in a hot path) OR 3+ medium findings.
   - 30–49: multiple high-impact findings; production at scale would be unusable.
   - 0–29: pervasive inefficiency; the code cannot serve realistic load.
4. Cap findings at 8. Do not pad. Quality over quantity.
5. When you have nothing concrete to flag, leave the array empty.
